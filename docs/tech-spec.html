1. Overview

PureFeed is a self-hosted, multi-user Telegram feed filter. Each user authenticates their own Telegram account as a userbot (Telethon + MTProto), configures independent feeds that map source channels to a destination channel with keyword-based ad filtering, and receives clean forwarded content — complete with working unread counters.

Key Design Decisions

DecisionRationale
No web UIAll management via Telegram inline keyboards — zero external port exposure needed beyond the health check
Per-user Telethon clientsEach user monitors only their own subscribed channels; complete data isolation by user_id
Relay Buffer channelBot token as the forwarder generates unread counters; direct userbot forwards do not
QR-code auth onlyTelegram blocks server-IP phone+code logins; QR auth completes on the user's device
SQLite (not Postgres)Zero-config, single file, sufficient for multi-user load with async aiosqlite
In-memory conversation stateBot dialogs are ephemeral; only persistent data goes to DB
FastAPI as shellClean asyncio lifespan hooks + Docker health-check endpoint; no REST API beyond /api/status

3. Architecture

Event-driven async service. Three independent async loops share one Python process:

  1. Telethon event loops — one TelegramClient per authenticated user; receives MTProto push events from Telegram
  2. Bot API polling loop — single getUpdates long-poll (30 s timeout); handles all bot interactions
  3. Uvicorn — ASGI server on port 8000 for the Docker health-check endpoint

Data Flow

Telegram Channel (source)
        │ MTProto push event
        ▼
TelegramClient (Telethon, per-user)
        │ NewMessage event
        ▼
_process_message()
        │ match chat.id → id_to_feeds (numeric, fast)
        │ mark source as read (send_read_acknowledge)
        ▼
    [grouped_id?]──yes──▶ buffer 1.0 s ──▶ _forward_media_group()
        │no                                       │
        ▼                                         ▼
_process_single()  ◀────────────────────────────────
        │ ad_filter.is_ad() → skip if ad
        ▼
_forward_message()
        ├── relay path (preferred)
        │     Telethon → relay channel (silent)
        │     Bot API forwardMessage → destination   ← unread counter ✅
        │     Telethon delete from relay
        │
        └── direct fallback (if relay fails)
              Telethon → destination                 ← no unread counter ❌

Write Path (user action)

User taps button → handle_callback() or handle_message()
→ DB write (aiosqlite)
→ bot.reload_feeds() → _setup_handler(resolve_ids=True)
→ _show(chat_id, ...) — delete old message, send new (scroll-to-bottom)

Read Path (monitoring startup)

start_monitoring():
  Phase 1: _setup_handler(resolve_ids=False)  ← register immediately
  iter_dialogs()                               ← populate entity cache (~1–2 s)
  Phase 2: _setup_handler(resolve_ids=True)   ← rebuild with numeric IDs
  _init_bot_relay()                            ← join + promote in relay channel
  asyncio.create_task(_updates_keepalive())   ← 30 s GetStateRequest loop

Error Handling Strategy

FailureResponse
Relay failsLog warning; fall back to direct userbot forward (message not lost)
FloodWaitErrorSleep e.seconds; no retry
Per-feed exceptionLog error; continue to next feed (non-fatal)
Auth exceptionSurface to user via bot message; other users unaffected
Polling exceptionLog; sleep 5 s; restart loop (never crashes)
Keepalive exceptionSilently swallowed; loop continues

7. Authentication & First-Launch Setup

QR Login Flow (recommended)

  1. User: /settings → enters api_id and api_hash via 🔑 buttons → both saved to user_settings
  2. User: presses 📷 Войти через QR-код
  3. Bot: calls bot.start_qr_auth() → client.qr_login() → gets tg://login?token=… URL
  4. Bot: generates QR PNG via qrcode library → sends via sendPhoto with 2-minute notice
  5. Bot: creates background task _qr_auth_waiter(chat_id, bot, sm); sets state auth_qr_wait (stores task ref)
  6. User: opens Telegram → Settings → Devices → Scan QR code (approval happens device-side)
  7. Background task awaits qr.wait(timeout=120 s):
    • Success → clear state → bot.start_monitoring() → show feed list
    • Timeout → notify "QR-код истёк. Попробуйте ещё раз"
    • SessionPasswordNeededError → transition to auth_2fa_qr state
  8. [If 2FA] User types password → bot.confirm_2fa(password) → start_monitoring()
Why QR, not phone+code? Telegram's anti-phishing system blocks login attempts from data-center IPs. QR auth happens on the user's device — the server IP is never part of the authorization handshake.

Phone Login Flow (alternative)

  1. Steps 1 same as above (api_id + api_hash entry)
  2. User: presses 📱 Войти по телефону → enters +49XXXXXXXXX (with country code)
  3. Bot: client.send_code_request(phone) → stores phone_code_hash in state ctx → state auth_code
  4. User: enters 5-digit code from Telegram app
  5. Bot: confirm_auth(phone, code, hash) — possible exceptions: PhoneCodeInvalidError, PhoneCodeExpiredError, SessionPasswordNeededError
  6. [If 2FA] State auth_2fa → user enters password

Session Persistence

After auth, Telethon writes /app/data/telegram_{user_id}.session. On container restart, start_all() reloads credentials from user_settings and reconnects using the existing session file — no re-auth required.

Message Dispatch

_process_message(event, channel_to_feeds, id_to_feeds):
    chat_id = event.chat.id  # positive Telethon ID

    if chat_id in id_to_feeds:          # Fast path (numeric, 100% reliable)
        feeds = id_to_feeds[chat_id]
    else:                               # Fallback (string key)
        key = f"@{chat.username}" or f"-100{chat_id}"
        feeds = channel_to_feeds.get(key)
        if not feeds: log.debug("Не в фидах"); return

    # Mark source as read (best-effort)
    try: await client.send_read_acknowledge(chat, message)
    except: pass

    if message.grouped_id:
        await _buffer_media_group(message, source_name, feeds)
    else:
        await _process_single(message, source_name, feeds)

Media Group Buffering

_buffer_media_group(message, source_name, feeds):
    gid = message.grouped_id
    _media_groups[gid]["messages"].append(message)

    if gid in _media_group_tasks:
        _media_group_tasks[gid].cancel()          # reset timer

    _media_group_tasks[gid] = asyncio.create_task(
        _process_media_group_later(gid)           # new 1.0 s timer
    )

_process_media_group_later(gid):
    await asyncio.sleep(1.0)
    group = _media_groups.pop(gid)
    messages = sorted(group["messages"], key=lambda m: m.id)
    text = next((m.text for m in messages if m.text), "")
    for feed in group["feeds"]:
        is_ad, reason = await ad_filter.is_ad(text, feed["keywords"])
        if is_ad: log.info("[{feed}] Реклама (альбом N шт.)"); continue
        await _forward_media_group(messages, feed["destination_channel"])

Keepalive

_updates_keepalive():                          # started as asyncio.Task
    while self._monitoring:
        await asyncio.sleep(30)                # interval: 30 s
        try:
            await client(functions.updates.GetStateRequest())
        except Exception:
            pass                               # silently ignored
Without the keepalive, Telegram channels can delay push events by 10–15 minutes. GetStateRequest forces Telegram to flush any buffered updates.

9. Relay & Unread Counters

Telegram marks a message as unread only when it arrives from a bot token sender. The Relay Buffer channel bridges the userbot (which can read source channels) with the bot token (which generates unread counters).

Relay Channel Requirements

RequirementWho sets itHow
Bot token is adminService owner (once)Channel settings → Administrators → Add Admin, grant "Add Admins" right
Permanent invite linkService owner (once)Set RELAY_INVITE_LINK in .env
Each userbot is adminAutomatic_init_bot_relay() on start_monitoring()

Relay Init Sequence

_init_bot_relay():
    if not BOT_TOKEN: warn + return
    if not RELAY_CHANNEL_ID: warn with instructions + return

    # 1. Join relay channel (no-op if already member)
    if RELAY_INVITE_LINK:
        try: await client.join_channel(RELAY_INVITE_LINK)
        except: pass

    # 2. Resolve entity and calculate canonical Bot API chat_id
    entity = await _resolve_entity_telethon(RELAY_CHANNEL_ID)
    full   = await client.get_entity(entity)
    relay_chat_id = -(1000000000000 + full.id)

    # 3. Promote userbot as admin
    POST /promoteChatMember {
        chat_id: relay_chat_id, user_id: _userbot_id,
        can_post_messages: True, can_edit_messages: True, can_delete_messages: True
    }
    if "owner" in error: pass   # already owner — ignore

    log("Relay готов: канал {relay_chat_id} «{title}»")

Single Message Relay

# Step 1: userbot → relay (silent, no notification)
relay_msgs = await client.forward_messages(relay_entity, [msg], silent=True)
relay_id   = relay_msgs[0].id

# Step 2: bot token → destination (generates unread counter)
POST /forwardMessage {
    chat_id: destination, from_chat_id: relay_chat_id, message_id: relay_id
}  # timeout: 15.0 s

# Step 3: cleanup relay
await client.delete_messages(relay_entity, [relay_id])

Album Relay

# Same as above, but plural:
relay_msgs = await client.forward_messages(relay_entity, messages, silent=True)
relay_ids  = [m.id for m in relay_msgs]

POST /forwardMessages {
    chat_id: destination, from_chat_id: relay_chat_id, message_ids: relay_ids
}
await client.delete_messages(relay_entity, relay_ids)
Fallback: Any exception in the relay path triggers a direct userbot forward (client.forward_messages(destination, messages)). The message is delivered but without an unread counter.

10. Filtering Engine

File: app/filter.py — class AdFilter, global singleton ad_filter.

Algorithm

is_ad(text, keywords, use_ai=False) → (bool, str):
    if not text or not text.strip():
        return (False, "")          # empty / media-only → always pass

    text_lower = text.lower()

    for rule in keywords:
        parts = [p.strip() for p in rule.split("+") if p.strip()]
        # "скидка+купить" → ["скидка", "купить"]
        # "реклама"       → ["реклама"]

        if all(part in text_lower for part in parts):
            label = " + ".join(parts) if len(parts) > 1 else parts[0]
            return (True, f"Найдено правило: '{label}'")

Rule Syntax

RuleLogicMatches example
рекламаsubstring, case-insensitive"Это РЕКЛАМА нашего партнёра"
скидка+купитьAND — both substrings present"Скидка 20%, купить сейчас!"
промокод+10%AND"Промокод на скидку 10% ..."
Each keyword rule is an independent OR condition. Within a rule, +-separated parts use AND logic. No wildcard/regex support. Substring match only (no word boundaries).

12. Key Algorithms

Channel Normalization

normalize_channel(channel: str) -> str:
    ch = channel.strip()

    # Invite links — keep unchanged
    if "t.me/+" in ch or "joinchat" in ch:
        return ch

    # Private channel: https://t.me/c/1812695632/5 → -1001812695632
    if "t.me/c/" in ch:
        parts = ch.split("t.me/c/")[-1].strip("/").split("/")
        if parts[0].isdigit():
            return f"-100{parts[0]}"
        return ch

    # Public URL: https://t.me/name → @name
    if "t.me/" in ch:
        ch = ch.split("t.me/")[-1].strip("/")

    # Already numeric ID (strip accidental @)
    if ch.lstrip("@").lstrip("-").isdigit():
        return ch.lstrip("@")

    # Add @ prefix
    if ch and not ch.startswith("@"):
        ch = "@" + ch

    return ch

Bot API Long-Poll Loop

run_polling(sm):
    offset = 0
    log("Команды бота: polling запущен")
    while True:
        try:
            resp = POST /getUpdates {
                offset: offset, timeout: 30, limit: 100
            }  # http_client timeout: 35.0 s
            for update in resp["result"]:
                offset = update["update_id"] + 1
                try:
                    if "message":        await handle_message(chat_id, msg, sm)
                    if "callback_query": await handle_callback(cb, sm)
                except Exception as e:
                    log.error(f"Ошибка обработки update {id}: {e}")
        except asyncio.CancelledError:
            log.error("Bot commands polling остановлен"); return
        except Exception as e:
            log.warning(f"Polling ошибка: {e}")
            await asyncio.sleep(5)   # backoff before retry

Relay Buffer Channel Setup (one-time)

  1. Create a new private Telegram channel (e.g., "PureFeed Relay")
  2. Add the bot as administrator with all rights, including "Add Admins"
  3. Copy the channel ID (forward any message from it to @userinfobot)
  4. Create a permanent invite link with no expiry and no user limit

Service Deployment

git clone git@github.com:JuliaSivridi/PureFeed.git
cd PureFeed
cp .env.example .env
nano .env
# Fill in:
#   TELEGRAM_BOT_TOKEN=...
#   RELAY_CHANNEL_ID=-100...
#   RELAY_INVITE_LINK=https://t.me/+...

mkdir -p data
chmod 777 data/           # Docker container runs as root; dir must be writable

docker compose up -d
docker compose logs -f    # Confirm: "База данных инициализирована", "polling запущен"

First User Setup (in Telegram)

  1. Find the bot → /settings
  2. Enter api_id from my.telegram.org
  3. Enter api_hash from my.telegram.org
  4. Press 📷 Войти через QR-код
  5. Open Telegram → Settings → Devices → Scan QR code
  6. After auth: /feedlist → create feeds → add source channels → Start monitoring
